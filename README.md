# doc-to-audio

A production-ready monorepo that turns documents (TXT, Markdown, PDF, DOCX) into
audio you can stream and listen to anywhere. Upload a file, queue a conversion,
and play back the generated MP3 with resume-where-you-left-off support.

## Stack

| Workspace        | Tech                                                              |
| ---------------- | ---------------------------------------------------------------- |
| `apps/web`       | Next.js 14 (App Router), TypeScript, Tailwind, shadcn/ui          |
| `apps/api`       | Node.js, Fastify, TypeScript, Prisma (PostgreSQL), BullMQ (Redis) |
| `packages/types` | Shared TypeScript interfaces consumed by both apps               |

Audio is generated by a BullMQ **worker** that extracts text, optionally cleans
it with **Google Gemini**, and synthesizes speech with **Google Cloud
Text-to-Speech**.

```
┌────────────┐   POST /jobs   ┌────────────┐   enqueue    ┌──────────┐
│  Next.js   │ ─────────────▶ │  Fastify   │ ───────────▶ │  Redis   │
│   (web)    │                │   (api)    │              │ (BullMQ) │
└────────────┘                └────────────┘              └────┬─────┘
      ▲                             │                          │
      │  GET /api/audio/:jobId      │ Prisma                   │ consume
      │  (range-streamed MP3)       ▼                          ▼
      │                       ┌────────────┐            ┌────────────┐
      └────────────────────── │ PostgreSQL │            │   worker   │
                              └────────────┘            │ TTS + MP3  │
                                                        └────────────┘
```

## Quick start (local)

### Prerequisites

- Node.js >= 20 and `pnpm` (`corepack enable`)
- PostgreSQL and Redis (or just run them via Docker — see below)
- A Google Cloud service-account JSON with **Text-to-Speech** enabled
- (Optional) a Gemini API key for text cleanup

### 1. Install

```bash
pnpm install
pnpm --filter @doc-to-audio/types build   # build the shared types once
```

### 2. Configure environment

```bash
cp apps/api/.env.example apps/api/.env
cp apps/web/.env.example apps/web/.env
# edit the values — at minimum set JWT_SECRET, DATABASE_URL, REDIS_URL,
# and GOOGLE_APPLICATION_CREDENTIALS
```

Place your Google credentials JSON somewhere and point
`GOOGLE_APPLICATION_CREDENTIALS` at it (e.g. `./gcp-credentials.json`).

### 3. Start Postgres + Redis (optional helper)

```bash
docker compose up -d postgres redis
```

### 4. Set up the database

```bash
pnpm db:generate          # prisma generate
pnpm --filter @doc-to-audio/api db:migrate:dev   # create + apply the first migration
# or, without migration files:
pnpm db:push
```

### 5. Run everything in dev

```bash
# In separate terminals (or use `pnpm dev` to run web + api together):
pnpm --filter @doc-to-audio/api dev       # API on :4000
pnpm --filter @doc-to-audio/api worker    # BullMQ worker
pnpm --filter @doc-to-audio/web dev       # Web on :3000
```

Open http://localhost:3000, register an account, upload a document, hit
**Convert**, and you'll be taken to the player which polls until the audio is
ready.

## Run the whole stack with Docker

```bash
# Provide a credentials file at the repo root (bind-mounted into api + worker):
cp /path/to/your-service-account.json ./gcp-credentials.json

# Optionally export secrets so compose can pick them up:
export JWT_SECRET="$(openssl rand -hex 32)"
export GEMINI_API_KEY="..."

docker compose up --build
```

Services:

| Service    | Port | Description                          |
| ---------- | ---- | ------------------------------------ |
| `web`      | 3000 | Next.js frontend                     |
| `api`      | 4000 | Fastify HTTP API (runs `db push`)    |
| `worker`   | —    | BullMQ worker (no exposed port)      |
| `postgres` | 5432 | PostgreSQL 16                        |
| `redis`    | 6379 | Redis 7                              |

> The `api` and `worker` containers bind-mount `./gcp-credentials.json`. Create
> that file before `docker compose up`, otherwise Docker will create a directory
> in its place and TTS will fail.

## API reference

All routes except `/auth/*` and `/health` require an `Authorization: Bearer <jwt>`
header.

| Method | Path                    | Description                                  |
| ------ | ----------------------- | -------------------------------------------- |
| POST   | `/auth/register`        | Create a user, returns `{ token, user }`     |
| POST   | `/auth/login`           | Verify credentials, returns `{ token, user }`|
| GET    | `/files`                | Folder + file tree for the user              |
| POST   | `/files/upload`         | Multipart upload (`file`, optional `folderId`)|
| DELETE | `/files/:id`            | Delete a file + its audio                    |
| POST   | `/folders`              | Create a folder (`name`, optional `parentId`)|
| DELETE | `/folders/:id`          | Recursively delete a folder                  |
| POST   | `/jobs`                 | Enqueue audio generation (`fileId`)          |
| GET    | `/jobs/:id`             | Job status + audio path                      |
| PATCH  | `/jobs/:id/position`    | Save playback position (`lastPosition`)      |
| GET    | `/audio/:jobId`         | Stream the MP3 (HTTP range supported)        |

## Notes on auth

The web app stores the API-issued JWT in an **httpOnly** cookie. Browser-facing
requests (status polling, audio streaming) go through thin Next.js route handlers
that read the cookie server-side and forward a Bearer token to the API, so the
token is never exposed to client JavaScript. Mutations use Next.js **server
actions**.
